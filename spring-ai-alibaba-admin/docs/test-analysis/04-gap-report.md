## P0 gaps (≤20, on critical paths — add these first)

### P0-1: `PromptController.*` — integration
- **Critical path**: CP-1 (Prompt version publish — Nacos push)
- **Reason to add**: CP-1 matches rubrics 2 (multi-table) and 5 (Nacos external system). A regression in the entrypoint silently breaks downstream agent apps that subscribe to Nacos for prompt config. `CLAUDE.md` explicitly forbids changes to the `POST /api/prompts/search` path because community SDKs depend on it — the whole prompt surface is high-blast-radius.
- **Test type**: integration
- **What to assert**: `POST /api/prompt/version` with a valid `PromptVersionCreateRequest` returns `Result<PromptVersion>` with `code=0`; `GET /api/prompt/version?promptKey=…&version=…` returns the new version; `Result.code` is **integer** (`0`), not string (per the wrapper-split note in `CLAUDE.md`).
- **Heuristic fired**: rule 1 (controller → integration).

### P0-2: `NacosClientService.*` — integration
- **Critical path**: CP-1 (Prompt version publish — Nacos push)
- **Reason to add**: Owns the actual `configService.publishConfig(dataId, group, jsonPayload)` call (`PromptVersionServiceImpl.java:131`) — the external boundary on which the whole prompt-distribution contract depends. Silent corruption here means downstream agents pick up the wrong prompt body without any visible error in admin.
- **Test type**: integration
- **What to assert**: Given a `Prompt` + `PromptVersion`, the published Nacos `dataId` follows the expected naming scheme; the payload deserializes back into the same `Prompt` object; publish failure surfaces as a non-zero `Result.code` (not swallowed).
- **Heuristic fired**: rule 2 (service name matches `*Nacos*` / `*Client*` → integration).
